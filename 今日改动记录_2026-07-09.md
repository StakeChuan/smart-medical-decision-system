# 智慧医疗辅助决策系统今日改动记录

日期：2026-07-09

## 一、今日主要解决的问题

### 1. 登录页默认显示账号和密码

**问题表现**

进入登录页时，账号和密码输入框会自动出现默认内容，看起来像系统保存了密码，不利于展示和安全性。

**修改方式**

- 移除登录页中写死的默认账号和密码。
- 新增本地存储键 `smartMedicalLastUsername`，只保存上一次成功登录的用户名。
- 登录成功后保存用户名，退出登录后仍保留用户名。
- 密码框每次进入登录页都保持为空。
- 调整输入框自动填充属性：
  - 用户名：`autocomplete="username"`
  - 密码：`autocomplete="new-password"` 或关闭自动保存倾向。

**涉及文件**

- `frontend/index.html`
- `frontend/app.js`

---

### 2. 手机端访问和页面适配

**问题表现**

原本前端请求地址固定使用 `127.0.0.1:8000`，手机访问电脑网页时，手机会把 `127.0.0.1` 理解成手机自己，导致无法连接后端。

**修改方式**

- 前端 API 地址改为动态识别当前访问主机。
- 如果是本机直接打开文件，则继续使用 `http://127.0.0.1:8000`。
- 如果通过局域网 IP 访问前端，则自动请求同一 IP 下的 `8000` 后端端口。
- 补充移动端样式，让列表、按钮、聊天窗口、报告区域在手机端不挤压。

**手机访问方式**

后端启动：

```bat
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

前端启动：

```bat
python -m http.server 5500 --bind 0.0.0.0
```

手机访问：

```text
http://电脑IPv4地址:5500
```

**涉及文件**

- `frontend/app.js`
- `frontend/styles.css`

### 3. AI 报告无法正确识别患者性别

**问题表现**

界面上患者显示为男性，但 AI 报告中出现“女性”或“性别未提供”等描述。

**原因分析**

原来的 AI 提示词主要传入问诊内容，没有稳定传入患者基础信息。大模型只能根据主诉、症状等文本猜测性别，容易误判。

**修改方式**

- 生成 AI 报告时，后端从问诊记录关联患者信息。
- 将患者基础信息写入大模型提示词：
  - 患者姓名
  - 性别
  - 年龄
  - 既往病史
  - 过敏史
- 要求模型基于这些结构化信息生成报告，减少性别、年龄等基础信息误判。

**涉及文件**

- `backend/app/main.py`

### 9. 新医生新增患者时出现 `[object Object]`

**问题表现**

新增医生账号登录后，在患者管理中新增患者时，前端 toast 显示 `[object Object]`，无法看懂具体错误。

**原因分析**

后端返回的是 FastAPI / Pydantic 的结构化校验错误数组，前端直接把对象转成字符串，导致显示 `[object Object]`。

**修改方式**

- 前端新增统一错误格式化方法。
- 支持解析：
  - 字符串错误
  - Pydantic 校验错误数组
  - 对象错误
- 患者电话格式增加前端预校验，减少无意义请求。

**涉及文件**

- `frontend/app.js`

---

### 10. 管理员医生管理增强：编辑医生、重置密码、禁用账号

**问题表现**

管理员只能新增医生，不能维护医生账号；如果医生账号需要修改姓名、改密码或停用，没有后台入口。

**修改方式**

- 数据库 `users` 表新增 `is_active` 字段。
- 管理员医生列表新增账号状态：
  - 启用
  - 已禁用
- 医生列表每行新增操作：
  - 查看患者
  - 编辑
  - 重置密码
  - 禁用 / 启用
- 禁用医生后：
  - 医生不能重新登录。
  - 已登录 token 后续请求也会被后端拦截。
  - 医生历史患者、问诊、AI 报告、消息记录不删除。
- 管理员不能禁用当前登录账号。
- 修复消息权限相关的中文乱码提示。

**涉及文件**

- `backend/app/models.py`
- `backend/app/schemas.py`
- `backend/app/main.py`
- `frontend/app.js`
- `frontend/styles.css`
- `migrate_user_active_status.sql`

## 二、今日新增或变更的数据表

### 1. `message_conversations`

用于保存两个用户之间的一组会话关系。

主要字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 会话 ID |
| `user_one_id` | 会话用户 1 |
| `user_two_id` | 会话用户 2 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

用途：

- 管理员和医生之间聊天
- 医生和医生之间聊天
- 避免两个人之间重复创建多个会话

---

### 2. `messages`

用于保存具体消息内容。

主要字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 消息 ID |
| `conversation_id` | 所属会话 ID |
| `sender_id` | 发送者 ID |
| `receiver_id` | 接收者 ID |
| `content` | 消息内容 |
| `is_read` | 是否已读 |
| `created_at` | 发送时间 |

用途：

- 保存聊天消息
- 统计未读消息
- 支持会话历史展示

---


### 8. 消息中心接口

#### 查询可聊天用户

```http
GET /messages/users
```

作用：

- 管理员可看到医生。
- 医生可看到管理员和其他医生。

#### 查询会话列表

```http
GET /messages/conversations
```

作用：

- 返回当前用户所有会话。
- 包含最后一条消息、最后消息时间、未读数量。

#### 查询会话消息

```http
GET /messages/conversations/{conversation_id}
```

作用：

- 返回某个会话中的消息历史。

#### 发送消息

```http
POST /messages
```

请求示例：

```json
{
  "接收者ID": 2,
  "消息内容": "请查看这个患者的问诊记录"
}
```

#### 手动收取消息

```http
POST /messages/fetch
```

作用：

- 手动拉取未读消息。
- 有新消息则提示数量。
- 没有新消息则提示暂无新消息。

#### 标记会话已读

```http
POST /messages/conversations/{conversation_id}/read
```

作用：

- 将当前会话中发给自己的消息标记为已读。
